## Инженерный аудит: API и безопасность

### CRITICAL

#### 1. Отсутствие Rate Limiting на API
Ни один эндпоинт не имеет rate limiting. Атакующий может:
- Брутфорсить промокоды через `/api/promo/validate`
- Спамить создание заказов через `/api/orders`
- DDoS-ить чат-поллинг

**Рекомендация:** Добавить `express-rate-limit` или аналог на все публичные эндпоинты.

#### 2. JWT-токены без ротации и отзыва
Токены не имеют механизма отзыва. Если токен утёк — он валиден до истечения срока. Refresh-token отсутствует.

**Рекомендация:** Внедрить refresh-token паттерн + blacklist для отозванных токенов.

#### 3. Файловые загрузки без валидации содержимого
Загрузка файлов проверяет только расширение и MIME-тип из заголовка, но не содержимое файла. Возможна загрузка вредоносных файлов с подменённым расширением.

**Рекомендация:** Валидировать magic bytes файлов, сканировать на вредоносное содержимое.

#### 4. Telegram initData валидация неполная
Проверяется hash, но не проверяется `auth_date` (давность). Перехваченный initData может быть переиспользован без ограничения по времени.

**Рекомендация:** Проверять `auth_date` и отклонять запросы старше 5 минут.

### HIGH

#### 5. SQL-инъекции через динамические запросы
В нескольких местах используется конкатенация строк для построения SQL-запросов (фильтрация заказов, поиск). Параметризация применяется непоследовательно.

#### 6. CORS настроен с wildcard в development
`cors({ origin: '*' })` в dev-режиме. Если development-переменные попадут в production — полный доступ ко всем эндпоинтам с любого домена.

#### 7. Отсутствие логирования безопасности
Нет аудит-лога для: входа администратора, изменения статусов заказов, платёжных операций. Невозможно расследовать инциденты.

#### 8. Секреты в environment без шифрования
BOT_TOKEN, JWT_SECRET, DATABASE_URL хранятся в plain text .env. Нет интеграции с secret manager.

#### 9. Отсутствие CSRF-защиты
Для cookie-based auth (admin panel) нет CSRF-токенов.

#### 10. WebSocket-соединения без аутентификации при reconnect
При переподключении WebSocket не перепроверяет валидность токена.

### MEDIUM

- Нет health check эндпоинта для мониторинга
- Ответы API раскрывают внутренние stack traces в production
- Нет pagination limit (можно запросить 100000 записей)
- Отсутствует Content-Security-Policy header
- Нет request validation middleware (zod/joi) для тел запросов
- Загруженные файлы доступны по прямой ссылке без авторизации
- Отсутствует graceful shutdown для сервера
- Нет compression middleware (gzip/brotli)

### LOW

- Swagger/OpenAPI документация отсутствует
- Нет request ID для трассировки
- Версионирование API отсутствует
- Дублирование middleware между роутерами
- Неконсистентный формат ответов (camelCase vs snake_case)
- Нет health-check для WebSocket
- Конфигурация TypeScript не строгая (`strict: false` в некоторых tsconfig)

## Инженерный аудит: Бот и база данных

### CRITICAL

#### 1. God Mode без аутентификации
`GodModePage` доступна через URL без дополнительной проверки прав. Любой пользователь, знающий URL, получает admin-доступ.

**Рекомендация:** Серверная проверка admin-роли на каждом запросе God Mode.

#### 2. Платёжные операции без транзакций
Обновление статуса заказа и начисление средств выполняются отдельными запросами без SQL-транзакции. При сбое между операциями — рассогласование данных.

**Рекомендация:** Обернуть в `BEGIN...COMMIT` транзакции.

#### 3. Нет миграций базы данных
Схема применяется через единый SQL-файл. Нет системы миграций, нет версионирования схемы, откат невозможен.

**Рекомендация:** Внедрить Knex migrations или Prisma.

#### 4. Отсутствие индексов на критичных столбцах
`orders.user_id`, `orders.status`, `messages.order_id`, `transactions.user_id` — нет индексов. Запросы к большим таблицам будут деградировать.

#### 5. Бот не обрабатывает ошибки Telegram API
Вызовы `bot.telegram.sendMessage()` не обёрнуты в try/catch. Ошибка отправки одного сообщения роняет весь обработчик.

#### 6. Реферальная система уязвима к self-referral
Пользователь может использовать собственный реферальный код. Нет проверки `referrer_id !== user_id`.

#### 7. Пул соединений БД не настроен
Используется единственное соединение к PostgreSQL. При нагрузке — bottleneck и таймауты.

#### 8. Нет бэкапов базы данных
Отсутствует cron-задача или скрипт для бэкапов. Потеря данных необратима.

### HIGH

- Клубные баллы начисляются без верхнего лимита — возможна эксплуатация
- Промокоды не имеют лимита использований per-user
- Статусы заказов можно менять в произвольном порядке (нет state machine)
- Бот не валидирует callback_data — можно отправить произвольные данные
- Нет graceful shutdown для бота (webhook vs polling конфликт)
- Дублирование бизнес-логики между ботом и API (расчёт цен)
- `user_data` хранится JSON-полем без схемы — нет валидации
- Уведомления пользователям жёстко зашиты в коде бота (не шаблонизированы)
- Нет retry-логики для исходящих запросов бота
- Chat messages не шифруются в БД
- Нет soft delete — удалённые заказы пропадают навсегда
- Файлы заказов хранятся на файловой системе сервера (не S3/CDN)
- Промокод валидируется на клиенте, но не перепроверяется при создании заказа
- Cron-задачи для очистки устаревших данных отсутствуют

### MEDIUM

- Telegram webhook secret не верифицируется
- Нет логирования действий бота (structured logging)
- Кэширование отсутствует (Redis не используется)
- Бот не поддерживает graceful degradation при недоступности БД
- Нет health-check для бота
- Сессии пользователей не инвалидируются при бане
- Нет мониторинга длины очереди сообщений
- Тесты отсутствуют полностью
- Нет seed-данных для разработки
- Дублирование констант между backend и mini-app
- Нет механизма feature flags

### LOW

- Логи бота пишутся в stdout без ротации
- Нет Dockerfile для бота (только для API)
- README не описывает архитектуру бота
- Конфигурация бота частично hardcoded
- Нет метрик (Prometheus/Grafana)
- Типы ошибок не стандартизированы
- Нет changelog
- Зависимости не зафиксированы (нет lock-файла для бота)
- Нет pre-commit hooks

## UX/UI аудит: Mini App

### CRITICAL

#### 1. OrderDetailPageV8.tsx — монолитный компонент (1600+ строк)
Содержит: дизайн-систему (`DS`), `STATUS_CONFIG`, утилиты, хук `usePaymentCountdown`, `OrderAppBar`, `HeroSummary`, PaymentSheet, ConfirmPaymentModal, TrustSection, FilesSection. Должен быть разбит на 8-10 файлов.

#### 2. Хаос стилизации: inline styles + Tailwind + CSS Modules
Три несовместимых подхода в одном проекте без чётких границ. `OrderDetailPageV8.tsx` использует все три одновременно. `ClubPage.tsx` импортирует стили из `HomePage.module.css`.

### HIGH

#### 3. Хардкод цветов вместо CSS-переменных
Несмотря на систему переменных (`--gold-400`, `--text-primary`), компоненты используют raw RGBA:
- `HomeHeader.tsx`: `'rgba(212, 175, 55, 0.72)'`
- `ActiveOrderDashboard.tsx`: `'rgba(212, 175, 55, 0.18)'`
- `BatchPaymentPage.tsx`: `'rgba(59,130,246,0.15)'`

#### 4. Светлая тема сломана
`ThemeContext.tsx` поддерживает light/dark, но 90%+ UI захардкожен под тёмную тему. Переключение создаёт нечитаемый интерфейс.

#### 5. Чат: polling + WebSocket дублирование
`OrderChatPage.tsx` использует `setInterval(loadMessages, 5000)` параллельно с WebSocket. При получении WS-сообщения делает полный HTTP-запрос — двойная нагрузка.

#### 6. Клавиатура не обрабатывается в чате
Textarea в `OrderChatPage.tsx` не учитывает overlay клавиатуры Telegram. Нет `visualViewport` listener.

#### 7. Низкий контраст текста (WCAG fail)
Текст с opacity 0.25-0.35 на тёмном фоне даёт контраст ~1.3:1 (требуется 4.5:1).

#### 8. Дублирование `formatMoney` в 5+ файлах
Каждая реализация немного отличается (`Math.max` vs `Math.round`).

#### 9. Три разные системы статусов заказов
`OrdersPage` → `STATUS_META`, `OrderDetailPageV8` → `STATUS_CONFIG`, `ActiveOrderDashboard` → `ORDER_STATUS_MAP`.

#### 10. Тяжёлая загрузка шрифтов (~500KB)
5 семейств шрифтов (Montserrat, Cinzel, Manrope, JetBrains Mono, Cormorant Garamond) + повторный импорт через CSS.

### MEDIUM

- `100vh` vs `100dvh` — несогласованность (страницы перекрываются на iOS)
- Нет Error Boundary на уровне страниц (кроме OrderDetail)
- Framer Motion на каждом элементе — jank на слабых устройствах
- `PremiumBackground` рендерится на 5 страницах (возможно наложение)
- Нет виртуализации длинных списков
- Prop drilling `user` объекта без Context
- WebSocket context пересоздаётся на каждое сообщение (перерендер всех consumers)
- `Navigation.tsx` scroll detection работает только на HomePage
- Touch targets < 44px (QuickActions 28px, FilterChip 36px)
- Отсутствуют ARIA labels (поиск, кнопки записи, навигация)
- `BatchPaymentPage` useEffect с пустыми зависимостями при зависимости от `searchParams`
- WelcomeTour принудительно перенаправляет на создание заказа

### LOW

- `EmptyStateOnboarding` — мёртвый код
- Telegram MainButton не используется (custom CTA вместо нативного)
- Source maps отключены в production
- Race condition при двойном тапе на submit в CreateOrderPage
- Нет retry-механизма на ClubPage при ошибке загрузки

## Контент-аудит

### CRITICAL

#### 1. Смешение языков без системы i18n
Интерфейс на русском, но код содержит английские строки (`"Loading..."`, `"Error"`, `"Submit"`), константы и placeholder-тексты на английском. Нет системы интернационализации.

**Рекомендация:** Внедрить `react-i18next` и вынести все строки в языковые файлы.

#### 2. Юридические тексты отсутствуют
Нет: Terms of Service, Privacy Policy, политики возврата, согласия на обработку персональных данных. Для платёжного сервиса это юридический риск.

### HIGH

#### 3. Тексты ошибок неинформативны
Пользователь видит: `"Ошибка загрузки"`, `"Что-то пошло не так"` без указания причины и действий. Нет guidance для пользователя.

#### 4. Inconsistent тон коммуникации
Микс формального и неформального стиля. Одни тексты на «вы» (`"Ваш заказ готов"`), другие на «ты» (`"Создай свой первый заказ"`).

#### 5. Placeholder-тексты в production
Найдены: `"Lorem ipsum"`, `"TODO"`, `"test"` в нескольких компонентах и константах.

#### 6. SEO полностью отсутствует
`index.html` имеет generic `<title>` и `<meta description>`. Нет OG-тегов, нет structured data. Для Mini App не критично, но веб-версия страдает.

#### 7. Уведомления бота — стена текста
Сообщения бота длинные, не используют Telegram Markdown formatting, нет структуры (заголовки, списки).

#### 8. Onboarding слишком агрессивный
Welcome Tour сразу ведёт к созданию заказа без объяснения что такое сервис, как он работает, сколько стоит.

### MEDIUM

- Описания статусов заказов непонятны пользователю (`"В работе"` — что именно делается?)
- FAQ/Help — минимальный контент, не покрывает частые вопросы
- Промо-тексты жёстко зашиты в код (не CMS)
- Empty states не мотивируют к действию
- Тексты кнопок неконсистентны (`"Отправить"` vs `"Создать"` vs `"Готово"`)
- Нет микрокопирайтинга в формах (подсказки, примеры)
- Числовые форматы не локализованы (точка vs запятая)

### LOW

- Нет changelog для пользователей (what's new)
- Описания в App Store / Telegram Bot не оптимизированы
- Нет tooltip-подсказок для иконок без текста
